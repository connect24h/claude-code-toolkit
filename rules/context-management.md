# コンテキスト管理ルール

## 原則

コンテキストは有限リソース。効率的に使用し、肥大化を防ぐ。

---

## 必須ワークフロー

### 大規模タスク（3ステップ以上）

```
1. Plan（計画）
   → 計画をファイルに保存
   → コンテキストには要約のみ

2. Design（設計）
   → タスクを分割
   → 依存関係を整理

3. Distribute（振り分け）
   → Codex: 単純タスク
   → Gemini: 分析タスク
   → Subagent: 中複雑タスク

4. Execute（実行）
   → 並列実行で時間短縮
   → 各タスクは独自コンテキスト

5. Integrate（統合）
   → 結果を要約して報告
```

---

## サブエージェント活用

### ✅ 推奨

```
Task({
  subagent_type: "Explore",
  prompt: "src/以下で認証関連のファイルを特定"
})
→ 親コンテキストを消費しない
→ 結果の要約のみ返る
```

### ❌ 非推奨

```
親コンテキストで:
- Read(file1)
- Read(file2)
- Read(file3)
- ...
→ コンテキストが肥大化
```

---

## CLI使い分け

| 条件 | 使用CLI | 理由 |
|------|---------|------|
| 1-2ファイル修正 | Codex | 高速、独立 |
| 大量ファイル分析 | Gemini | 200万トークン |
| 中複雑度タスク | Subagent | コンテキスト分離 |
| 判断が必要 | Claude本体 | 対話必要 |

---

## 結果の扱い

### ファイルに保存すべきもの

- 計画書
- 詳細な分析結果
- 生成したコード
- ログ

### コンテキストに残すもの

- 要約（1-3行）
- ステータス
- 次のアクション

---

## 並列実行

```
# 独立タスクは同時実行
Task({ task1 })  # 同時
Task({ task2 })  # 同時
Task({ task3 })  # 同時

# 依存タスクは順次
Task({ task1 })  # 先に
# 結果を待つ
Task({ task2_depends_on_1 })  # 後で
```

---

## アンチパターン

### ❌ 避けるべき

```
1. 全ファイルを親コンテキストで読む
2. サブエージェントの結果を全て展開
3. 計画なしで大規模実装
4. 単純タスクを親コンテキストで処理
5. 順次実行で時間浪費
```

### ✅ 推奨

```
1. Exploreエージェントでファイル探索
2. 結果は要約のみ受け取る
3. 必ず計画フェーズから開始
4. Codex/Geminiに単純タスク委譲
5. 並列実行を活用
```

---

## コンテキスト警告サイン

以下の場合、コンテキスト肥大化の可能性:

- 多数のファイルを直接Read
- 長い会話履歴
- サブエージェント結果の全展開
- 計画なしの大規模作業

**対処:** サブエージェントへの委譲、結果の要約、ファイル保存
