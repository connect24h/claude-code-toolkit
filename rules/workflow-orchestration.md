# ワークフローオーケストレーション

## 1. Plan Node Default

非自明なタスク（3ステップ以上 or アーキテクチャ判断を伴う）は**必ずプランモードで開始**する。

- 実装前に `tasks/todo.md` にチェック可能な計画を書く
- 計画をユーザーに確認してから実装開始
- **何かがうまくいかなくなったら即座に停止し、再計画する**（押し通さない）
- 検証ステップも計画に含める（ビルドだけでなく）
- 曖昧さを減らすため、事前に詳細な仕様を書く

## 2. サブエージェント戦略

メインコンテキストウィンドウを清潔に保つため、サブエージェントを積極的に使う。

- 調査、探索、並列分析はサブエージェントにオフロード
- 複雑な問題にはサブエージェント経由で計算リソースを投入
- **1タスク1サブエージェント**（集中実行）
- メインコンテキストには結果のサマリーのみ保持

## 3. 自己改善ループ

ユーザーからの**あらゆる修正の後**、パターンを記録する。

- `tasks/lessons.md` に修正パターンを追記
- 同じミスを防ぐルールを自分で書く
- ミス率が下がるまで徹底的にイテレーション
- セッション開始時に関連プロジェクトのlessonsを確認

### lessons.md フォーマット

```markdown
## [日付] [カテゴリ]
- **状況**: 何が起きたか
- **原因**: なぜ起きたか
- **教訓**: 次回どうするか
- **ルール**: 自動適用すべきルール
```

## 4. 完了前検証

タスクを完了とマークする前に**必ず動作を証明**する。

- テスト実行、ログ確認、正しさのデモンストレーション
- 必要に応じてmainブランチとの差分を確認
- 自問: **「シニアエンジニアがこれを承認するか？」**
- 証明なき完了は許容しない

## 5. エレガンスの追求（バランス型）

非自明な変更に対してのみ適用。単純な修正には過剰適用しない。

- 非自明な変更時: 「もっとエレガントな方法はないか？」と立ち止まる
- ハック的な修正: 「今知っていることを全て踏まえ、エレガントな解決策を実装する」
- 単純で明白な修正にはスキップ（過剰エンジニアリング禁止）
- 提出前に自分の仕事を批判的に見る

## 6. 自律的バグ修正

バグ報告を受けたら、手取り足取り聞かずに**自分で修正する**。

- ログ、エラー、失敗テストを指し示し、自ら解決
- ユーザーのコンテキストスイッチをゼロにする
- CIの失敗テストも指示なしで修正に向かう
- 根本原因を特定し、一時的な修正ではなく本質的な修正を行う
